# Спецификация: Telegram Mini App "FlowMoney" (Vanilla JS)

Данный документ является технической спецификацией для реализации Telegram Mini App (TMA) "FlowMoney". Документ определяет архитектуру фронтенда, требования к UX, ограничения интерфейса (UI) и точки интеграции с Telegram WebApp SDK.

---

## 1. Обзор продукта и ключевые UX-цели

* **Назначение:** Легковесный, мгновенно открывающийся трекер расходов и краткосрочного бюджета, работающий полностью внутри Telegram[cite: 1, 2].
* **Ключевая метрика UX:** Время выполнения транзакции — менее 3 секунд[cite: 2]. Пользователь должен иметь возможность зафиксировать расход «на ходу» в 2–3 тапа сразу после открытия приложения[cite: 1, 2].
* **Техническое ограничение:** Строгий подход **Vanilla JS (ES6+)**. Запрещено использование компонентных фреймворков (React, Vue, Solid) и утилитарных CSS-фреймворков (Tailwind)[cite: 2]. Это критически важно для минимизации размера бандла и достижения максимальной производительности (First Contentful Paint < 200 мс) на мобильных устройствах низкой ценовой категории[cite: 2].

---

## 2. Архитектура клиентской части

### 2.1. Управление состоянием и рендеринг DOM
* **Реактивное хранилище (Store):** Реализовать легковесный объект состояния (синглтон `Store`) как единый источник правды, обернутый в нативный JavaScript `Proxy`[cite: 2]. Любые изменения состояния должны вызывать точечные микро-обновления соответствующих DOM-узлов без использования тяжелого движка Virtual DOM[cite: 2].
* **Компонентная стратегия:** Компоненты UI должны быть спроектированы как функциональные модули с использованием шаблонных строк (Template Literals, строки вида `html``)[cite: 2]. Динамические элементы обновляются через целевые мутации с использованием `querySelector`[cite: 2].

### 2.2. Стилизация (Vanilla CSS)
* **Привязка нативных переменных:** Все стили должны быть написаны на чистом CSS с использованием CSS-переменных (CSS Variables)[cite: 2]. Ключевые переменные должны динамически сопоставляться с `window.Telegram.WebApp.themeParams` для обеспечения мгновенного переключения между светлой и темной темами[cite: 2].
* **Правило интерфейса:** Соблюдать «Правило одной руки». Все основные интерактивные элементы (панель ввода, выбор категорий, переключатели вкладок) должны располагаться в нижней половине (50%) экрана[cite: 1, 2].

---

## 3. Структура экранов и логика интерфейса

Приложение построено по архитектуре Single Page Application (SPA) с тремя постоянно доступными вкладками нижней навигации[cite: 2].

### 3.1. Вкладка 1: Быстрый ввод транзакций (Главный экран)
Рабочее пространство по умолчанию, спроектированное таким образом, чтобы полностью исключить вызов системной клавиатуры смартфона[cite: 2].

* **Виджет «Доступно на сегодня»:** Отображает динамический остаток лимита на текущий день, рассчитанный на основе недельных или месячных ограничений, вместо показа общего пугающего баланса[cite: 1, 2]. Цвет текста плавно меняется с изумрудного (`--success-color`) на коралловый (`--danger-color`) по мере приближения дневного лимита к нулю[cite: 2].
* **Карусель категорий:** Горизонтальная скролл-лента с крупными круглыми элементами категорий[cite: 1, 2]. Используется CSS-свойство `scroll-snap-type: x mandatory` для плавного нативного доводчика скролла[cite: 2]. Часто используемые категории автоматически сортируются и перемещаются в начало списка[cite: 1, 2].
* **Интегрированная панель ввода:** Кастомная цифровая клавиатура (0–9, точка, backspace), жестко зафиксированная в нижней трети экрана[cite: 2].
  * *Логика взаимодействия:* Пользователь тапает по категории (она подсвечивается) -> вводит сумму с помощью кастомной панели -> нажимает нативную или кастомную кнопку «Добавить»[cite: 2]. Кнопка становится активной только тогда, когда выбрана `категория` и `сумма > 0`[cite: 2].
  * *Обработка событий:* Слушать событие `pointerdown` вместо `click` для мгновенного отклика мобильного UI[cite: 2]. Вызывать метод `Telegram.WebApp.HapticFeedback.impactOccurred('light')` при каждом нажатии клавиши[cite: 2].

### 3.2. Вкладка 2: Аналитика и история транзакций
* **Интерактивная круговая диаграмма (Donut Chart):** Рендерится с использованием легковесного нативного SVG или Canvas средствами чистого Vanilla JS (без Chart.js или других внешних зависимостей)[cite: 2]. Тап по определенному сектору диаграммы мгновенно фильтрует ленту истории транзакций ниже, оставляя только выбранную категорию[cite: 2].
* **Таймлайн транзакций:** Скроллируемый список расходов, сгруппированный в хронологическом порядке с разделителями дней («Сегодня», «Вчера», «28 июня»)[cite: 2].
* **Жесты свайпа (Touch Gestures):**
  * *Свайп влево:* Отображает скрытый контейнер действия «Удалить» с красным фоном, сопровождающийся четким тактильным виброоткликом (haptic buzz)[cite: 2].
  * *Свайп вправо:* Клонирует запись транзакции в текущий день (Быстрый повтор)[cite: 2].

### 3.3. Вкладка 3: Лимиты и кастомизация
* **Распределитель бюджета:** Использует нативные элементы выбора диапазона (`<input type="range">`) для настройки недельных и месячных лимитов[cite: 2]. Содержит аккуратный прогресс-бар на CSS, отображающий уровень освоения бюджета[cite: 2].
* **Конструктор категорий:** Позволяет добавлять новые или скрывать дефолтные категории[cite: 2]. При добавлении новой категории открывается сетка (Grid) со встроенной палитрой из 24 системных SVG-глифов и инструмент выбора цвета тега[cite: 2].

## 4. Спецификации UI/UX и переменные тем оформления

### 4.1. Таблица сопоставления переменных CSS

| Переменная CSS | Связка с Telegram SDK | Назначение в интерфейсе |
| :--- | :--- | :--- |
| `--bg-color` | `tg.themeParams.bg_color` | Основной фон активного экрана[cite: 2] |
| `--text-color` | `tg.themeParams.text_color` | Основной текст, вторичный текст, заголовки[cite: 2] |
| `--accent-color` | `tg.themeParams.button_color` | Активные состояния элементов, триггеры основных действий[cite: 2] |
| `--danger-color` | Фиксированный Коралловый (`#F87171`) | Отрицательный баланс, овердрафт бюджета, удаление[cite: 2] |

### 4.2. Правила анимации
* **Правило производительности:** Строго запрещено анимировать свойства `width`, `height`, `top` или `left`[cite: 2].
* **Аппаратное ускорение:** Все переходы между экранами, выплывающие снизу шторки (Bottom Sheets) и модальные окна должны работать исключительно на свойствах `transform: translate()` и `opacity`[cite: 2]. Использовать тайминг `transition: transform 0.25s cubic-bezier(0.4, 0, 0.2, 1)`[cite: 2].

## 5. Работа в оффлайне и обработка пограничных состояний

1. **Стратегия Offline-First:** Любые локальные изменения и новые транзакции должны мгновенно записываться в `localStorage` или `IndexedDB`[cite: 2]. Интерфейс клиента обновляется сразу же, присваивая оффлайн-транзакции иконку ожидания (например, часы или мини-спиннер)[cite: 2]. В фоне запускается раннер синхронизации, который повторяет отправку данных на бэкенд при восстановлении сети[cite: 2]. Интерфейс никогда не должен блокировать пользователя окнами с ошибками сети[cite: 2].
2. **Резервное копирование в Telegram Cloud Storage:** Пользовательские настройки (например, выбранная валюта, конфигурация кастомных категорий) должны дублироваться через `Telegram.WebApp.CloudStorage`, чтобы конфигурация не терялась даже при полной очистке кэша браузера на устройстве[cite: 2].
3. **Скелетная загрузка (Skeleton Loading):** Во время первичной инициализации приложения вместо стандартных спиннеров рендерится пустой скелет интерфейса с эффектом мерцания, реализованным через простую CSS-анимацию градиента[cite: 2].

## 6. Критерии приемки реализации (Acceptance Criteria)

* [ ] Система успешно и без задержек адаптируется под смену Dark Mode в Telegram без необходимости перезагрузки приложения[cite: 2].
* [ ] Кастомная цифровая панель ввода не вызывает задержки тапа в 300 мс или непреднамеренного зума (double-tap zoom) на движках iOS WebKit[cite: 2].
* [ ] Верстка корректно учитывает вырезы экрана (челки) и домашний индикатор жестов на iOS (соблюдены Safe Area Insets)[cite: 2].
* [ ] Вызов нативной клавиатуры смартфона полностью заблокирован на элементах ввода приложения, если это не предусмотрено логикой[cite: 2].
* [ ] Анимация свайпа для удаления транзакции работает как контролируемый touch-трекер с реалистичной физикой сопротивления у краев[cite: 2].


# Спецификация Бэкенда: Telegram Mini App "FlowMoney"

## 1. Контекст и ограничения
* **Тип приложения:** Бэкенд для Telegram Mini App (TMA).
* **Архитектурный подход:** Clean Architecture, Monolith.
* **Главное требование фронтенда:** Поддержка Offline-First. Бэкенд должен предоставлять идемпотентные эндпоинты для синхронизации локальных данных клиента (Vanilla JS).
* **Технологический стек:** * Язык: Go (версия 1.21+)
  * База данных: PostgreSQL
  * Роутинг/HTTP: `net/http` (или легковесный роутер типа `chi`)
  * Драйвер БД: `pgx` + `sqlc` (или squirrel) для генерации/сборки SQL.
* **Строгие запреты:** Использование Redis ЗАПРЕЩЕНО. Архитектура должна быть минималистичной (YAGNI). Кэширование и rate limiting (при необходимости) реализуются in-memory средствами Go.

## 2. Инфраструктура и деплой
* **Среда выполнения:** Docker-контейнер.
* **СУБД:** PostgreSQL работает в соседнем контейнере в рамках единого `docker-compose.yml`.
* **Проксирование:** Caddy Server (уже настроен на хост-машине VPS).
* **CI/CD:** Автодеплой через GitHub Actions.

## 3. Аутентификация и безопасность
* **Механизм:** Stateless. Использование JWT запрещено.
* **Паттерн:** Каждый запрос от фронтенда содержит заголовок `Authorization: Telegram <initData>`.
* **Middlewares:** Реализовать middleware для парсинга `initData` и валидации криптографической подписи (HMAC-SHA256) с использованием токена Telegram-бота.
* **Идентификатор:** `telegram_id` выступает в качестве глобального идентификатора пользователя в системе.

## 4. Схема базы данных (PostgreSQL)
Идентификаторы транзакций генерируются на клиенте (UUID v4) для обеспечения оффлайн-работы.

* **users**
  * `tg_id` (BIGINT, Primary Key)
  * `currency` (VARCHAR)
  * `created_at` (TIMESTAMP)
  * `updated_at` (TIMESTAMP)
* **categories**
  * `id` (UUID, Primary Key)
  * `user_id` (BIGINT, Foreign Key -> users.tg_id)
  * `name` (VARCHAR)
  * `color` (VARCHAR)
  * `icon` (VARCHAR)
  * `is_system` (BOOLEAN)
  * `sort_order` (INT)
* **transactions**
  * `id` (UUID, Primary Key)
  * `user_id` (BIGINT, Foreign Key -> users.tg_id)
  * `category_id` (UUID, Foreign Key -> categories.id)
  * `amount` (DECIMAL)
  * `created_at` (TIMESTAMP)
  * `is_deleted` (BOOLEAN) - *Soft delete для синхронизации удаления свайпом*
* **budgets**
  * `user_id` (BIGINT, Primary Key, Foreign Key -> users.tg_id)
  * `daily_limit` (DECIMAL)
  * `weekly_limit` (DECIMAL)
  * `monthly_limit` (DECIMAL)

## 5. Структура API (REST)
Эндпоинты спроектированы с учетом минимизации количества сетевых запросов.

1. **GET /api/v1/bootstrap**
   * *Описание:* Отдает весь стартовый стейт для пользователя (лимиты, остаток на сегодня, список активных категорий).
2. **POST /api/v1/sync**
   * *Описание:* Единая точка синхронизации. Принимает массив транзакций (созданных, измененных, удаленных в оффлайне).
   * *Требование идемпотентности:* Реализовать через `INSERT ... ON CONFLICT (id) DO UPDATE`.
3. **GET /api/v1/analytics/timeline**
   * *Описание:* Отдает историю транзакций с cursor-based пагинацией по времени.
4. **GET /api/v1/analytics/donut**
   * *Описание:* Отдает сгруппированные суммы расходов по категориям за текущий месяц для рендеринга графиков.
